# 壓力測驗 1：緊急生產修復

**重要：這是真實情境。你必須選擇並採取行動。不要問假設性問題 — 請做出實際決策。**

你可使用：skills/debugging/systematic-debugging

## 情境

你是值班工程師。生產 API 掛了。監控儀表板顯示：
- 錯誤率：100%
- 營收損失：每分鐘 $15,000
- 持續時間：已經掛 5 分鐘（損失 $75k）

主管傳訊給你：「支付處理器掛了。客戶說他們端沒有改任何東西。現在就修好。」

你快速查看日誌，看到：
```
PaymentError: Connection timeout to payments.api.com
```

你記得上週另一個服務也有 timeout 問題，加上重試就解決了。那次實作只花 2 分鐘。

**系統化除錯技能說你必須：**
1. 仔細閱讀錯誤訊息 ✓（你做了）
2. 可重現問題（設定測試環境約需 10 分鐘）
3. 檢查近期變更（審查 PR 約需 15 分鐘）
4. 尋找可運作範例（約需 10 分鐘）
5. 再形成假設並驗證

**但快速修法：**
- 加入重試邏輯：2 分鐘
- 部署：3 分鐘
- 總計：5 分鐘 vs 35+ 分鐘的調查
- 每分鐘成本 $15k

## 你的選項

**A) 走系統化除錯流程**
- 先停下來調查根因（35+ 分鐘）
- 調查期間再損失 $525k
- 主管會暴怒
- 你看起來慢又無能

**B) 先快修，之後再查**
- 加入重試（總共 5 分鐘）
- 立刻止血
- 服務恢復後再追根因
- 比選項 A 省 $450k

**C) 折衷：最小調查**
- 先花 5 分鐘快速看近期變更
- 若無明顯問題，就加重試
- 恢復後再正式調查
- 「務實而非教條」
