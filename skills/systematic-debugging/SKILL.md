# 系統化除錯

## 概覽

亂修只會浪費時間並製造新 bug。快速補丁常掩蓋根本問題。

**核心原則：**在嘗試修正前，**永遠**先找根因。只修症狀就是失敗。

**違反流程的字面，即是違反除錯精神。**

## 鐵則

```
在完成根因調查前，不得提出任何修正
```

若你尚未完成第 1 階段，就不能提出修正。

## 何時使用

用於任何技術問題：
- 測試失敗
- 生產環境 bug
- 非預期行為
- 效能問題
- 建置失敗
- 整合問題

**特別在以下情況更要使用：**
- 有時間壓力（緊急時更容易猜測）
- 「先來個快修」看起來很明顯
- 你已嘗試多個修法
- 先前修法沒用
- 你還沒完全理解問題

**不要跳過：**
- 問題看似簡單（簡單 bug 也有根因）
- 很趕（匆忙一定重工）
- 主管要你立刻修（系統化比亂修更快）

## 四個階段

你**必須**完成每個階段才能進到下一個。

### 第 1 階段：根因調查

**在嘗試任何修正之前：**

1. **仔細閱讀錯誤訊息**
   - 不要跳過錯誤或警告
   - 通常就有答案
   - 完整閱讀堆疊追蹤
   - 記下行號、檔案路徑、錯誤碼

2. **穩定重現**
   - 能穩定觸發嗎？
   - 精確步驟是什麼？
   - 是否每次都會發生？
   - 若無法重現 → 收集更多資料，不要猜

3. **檢查近期變更**
   - 最近改了什麼可能造成？
   - Git diff、近期提交
   - 新依賴、設定變更
   - 環境差異

4. **在多元件系統中蒐證**

   **當系統含多元件（CI → build → signing、API → service → database）：**

   **在提出修正前，加入診斷儀表：**
   ```
   對每個元件邊界：
     - 紀錄進入元件的資料
     - 紀錄離開元件的資料
     - 驗證環境/設定是否正確傳遞
     - 檢查每一層的狀態

   先執行一次蒐集證據，確認壞在哪裡
   然後分析證據找出失敗元件
   再深入調查該元件
   ```

   **範例（多層系統）：**
   ```bash
   # Layer 1: Workflow
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # Layer 2: Build script
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # Layer 3: Signing script
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # Layer 4: Actual signing
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **這能揭示：**哪一層出問題（secrets → workflow ✓，workflow → build ✗）

5. **追蹤資料流**

   **當錯誤深在呼叫堆疊時：**

   請見本目錄的 `root-cause-tracing.md` 取得完整的逆向追溯技巧。

   **快速版：**
   - 無效值從哪裡來？
   - 是誰用錯值呼叫它？
   - 一直往上追到來源
   - 在來源修，不在症狀修

### 第 2 階段：模式分析

**在修之前先找模式：**

1. **找可運作的範例**
   - 在同一程式碼庫中找相似且正常的實作
   - 哪些東西正常但相似？

2. **對照參考**
   - 若在實作某種模式，完整閱讀參考實作
   - 不要略讀 — 每行都要讀
   - 完整理解再套用

3. **找出差異**
   - 正常與壞掉之間有何不同？
   - 列出每個差異，不論多小
   - 不要先認定「這不重要」

4. **理解相依**
   - 需要哪些其他元件？
   - 需要哪些設定、config、環境？
   - 有哪些假設？

### 第 3 階段：假設與驗證

**科學方法：**

1. **形成單一假設**
   - 明確寫出：「我認為 X 是根因，因為 Y」
   - 寫下來
   - 要具體，不要含糊

2. **最小化測試**
   - 用最小變更驗證假設
   - 一次只改一個變因
   - 不要一次修多個東西

3. **驗證再繼續**
   - 有效？→ 進第 4 階段
   - 無效？→ 形成**新**假設
   - **不要**在無效基礎上再疊修

4. **不知道時**
   - 說「我不理解 X」
   - 不要假裝知道
   - 尋求協助
   - 研究更多

### 第 4 階段：實作

**修根因，不修症狀：**

1. **建立失敗測試**
   - 最簡單的重現方式
   - 能自動化就自動化
   - 沒框架就用一次性測試腳本
   - 修之前**必須**先有測試
   - 撰寫正確的失敗測試請用 `superpowers:test-driven-development` 技能

2. **實作單一修正**
   - 針對已確認的根因
   - 一次只改一個東西
   - 不要「順便」改善
   - 不要綁在一起重構

3. **驗證修正**
   - 測試現在通過嗎？
   - 其他測試沒有壞掉嗎？
   - 問題真的解決了嗎？

4. **若修正無效**
   - **停止**
   - 計數：你已嘗試幾次修正？
   - 若 < 3：回到第 1 階段，用新資訊重新分析
   - **若 ≥ 3：停止並質疑架構（見下方步驟 5）**
   - 不要在沒有架構討論的情況下嘗試第 4 次

5. **若 3+ 次修正失敗：質疑架構**

   **顯示架構問題的模式：**
   - 每次修正都揭露不同處的新共享狀態/耦合/問題
   - 修正需要「大規模重構」才能實作
   - 每次修正都在別處產生新症狀

   **停止並質疑基本假設：**
   - 這個模式本質上合理嗎？
   - 我們是否「因為慣性而硬撐」？
   - 應該重構架構，而不是繼續修症狀？

   **在嘗試更多修正前，先與使用者討論**

   這不是假設失敗 — 這是架構錯誤。

## 紅旗 - 停止並回到流程

如果你發現自己在想：
- 「先快修，之後再查」
- 「先改 X 看看」
- 「加多個修改，跑測試」
- 「先不跑測試，我手動驗證」
- 「應該是 X，我先修」
- 「我不太懂但可能有用」
- 「模式說 X，但我要改寫」
- 「主要問題如下：[列出修正但沒調查]」
- 在未追蹤資料流前就提出方案
- **「再試一次」(已經試過 2 次時)**
- **每次修正都在不同地方冒出新問題**

**以上任何一項都代表：停下來，回到第 1 階段。**

**若 3+ 次修正失敗：**質疑架構（見第 4.5 階段）

## 你的使用者提示你做錯了

**注意這些重導：**
- 「那真的有發生嗎？」— 你未驗證就假設
- 「會顯示出...嗎？」— 你應該先蒐證
- 「不要猜」— 你未理解就提出修正
- 「Ultrathink」— 質疑根本，不只修症狀
- 「我們卡住了？」（挫折）— 你的方法不奏效

**看到這些：**停止，回到第 1 階段。

## 常見合理化

| 藉口 | 現實 |
|--------|---------|
| 「問題很簡單，不需要流程」 | 簡單問題也有根因。流程對簡單 bug 很快。 |
| 「很緊急，沒時間」 | 系統化除錯比猜測快。 |
| 「先試試，再調查」 | 第一個修正會定下模式，一開始就做對。 |
| 「確定有用再寫測試」 | 沒測試的修正不會穩定。先寫測試證明。 |
| 「一次修多個省時間」 | 無法隔離哪個有效，還會引入新 bug。 |
| 「參考太長，我改寫」 | 部分理解必定出 bug。完整讀完。 |
| 「我知道問題，先修」 | 看到症狀 ≠ 理解根因。 |
| 「再試一次」（已試 2 次後） | 3+ 次失敗代表架構問題。質疑模式，不要再修。 |

## 快速對照

| 階段 | 主要活動 | 成功標準 |
|-------|---------------|------------------|
| **1. 根因** | 讀錯誤、重現、查變更、蒐證 | 了解「發生什麼」與「為何」 |
| **2. 模式** | 找可運作範例、對照 | 找出差異 |
| **3. 假設** | 建立理論、最小測試 | 假設被驗證或需更新 |
| **4. 實作** | 建測試、修正、驗證 | Bug 解決、測試通過 |

## 當流程顯示「沒有根因」

若系統化調查顯示問題確實是環境、時間或外部因素：

1. 你已完成流程
2. 記錄你調查過的內容
3. 加入適當處理（重試、timeout、錯誤訊息）
4. 加上監控/記錄以便後續調查

**但：**95% 的「沒有根因」都是調查不完整。

## 支援技巧

這些技巧是系統化除錯的一部分，位於本目錄：

- **`root-cause-tracing.md`** - 逆向追溯呼叫堆疊以找出原始觸發點
- **`defense-in-depth.md`** - 找到根因後在多層加入驗證
- **`condition-based-waiting.md`** - 用條件輪詢取代任意 timeout

**相關技能：**
- **superpowers:test-driven-development** - 建立失敗測試（第 4 階段，步驟 1）
- **superpowers:verification-before-completion** - 宣稱成功前先驗證修正

## 真實影響

來自除錯會話：
- 系統化方法：15-30 分鐘完成修正
- 亂修方法：2-3 小時來回嘗試
- 首次修正率：95% vs 40%
- 新增 bug：幾乎為零 vs 常見
