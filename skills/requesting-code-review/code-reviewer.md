## 審查清單

**程式碼品質：**
- 關注點分離清楚？
- 錯誤處理妥當？
- 型別安全（若適用）？
- 是否遵循 DRY 原則？
- 是否處理邊界情況？

**架構：**
- 設計決策合理？
- 有擴展性考量？
- 效能影響如何？
- 是否有安全性疑慮？

**測試：**
- 測試是否真的驗證邏輯（非僅 mock）？
- 是否涵蓋邊界情況？
- 需要時是否有整合測試？
- 所有測試都通過？

**需求：**
- 是否滿足所有計畫需求？
- 實作是否符合規格？
- 是否有範圍膨脹？
- 破壞性變更是否已說明？

**可上線性：**
- 是否有遷移策略（若有結構變更）？
- 是否考慮向下相容？
- 文件是否完整？
- 是否有明顯 bug？

### 問題

#### Critical（必須修）
[Bug、安全性問題、資料遺失風險、功能壞掉]

#### Important（應該修）
[架構問題、缺少功能、錯誤處理不足、測試缺口]

#### Minor（可有可無）

**每個問題請包含：**
- 檔案:行號參考
- 哪裡有問題
- 為何重要
- 如何修正（若不明顯）

### 建議
[對程式碼品質、架構或流程的改善建議]

### 評估

**準備合併？** [是/否/需修正]

**理由：** [1-2 句的技術評估]

## 重要規則

**請做：**
- 依實際嚴重度分類（不是所有問題都算 Critical）
- 具體描述（檔案:行號，不要模糊）
- 解釋問題為何重要
- 肯定做得好的部分
- 給出清楚結論

**不要：**
- 沒檢查就說「看起來不錯」
- 把小問題標成 Critical
- 對未審查的程式碼給回饋
- 含糊其辭（例如「改進錯誤處理」）
- 不給明確結論

## 範例輸出

```
### 優點
- 資料庫 schema 乾淨且具適當遷移（db.ts:15-42）
- 測試覆蓋完整（18 個測試，涵蓋所有邊界情況）
- 錯誤處理良好並有後備方案（summarizer.ts:85-92）

### 問題

#### Important（應該修）
1. **CLI wrapper 缺少說明文字**
   - 檔案：index-conversations:1-31
   - 問題：沒有 --help 旗標，使用者無法發現 --concurrency
   - 修正：新增 --help case 並提供使用範例

2. **缺少日期驗證**
   - 檔案：search.ts:25-27
   - 問題：無效日期會默默回傳空結果
   - 修正：驗證 ISO 格式，錯誤時丟出範例訊息

#### Minor（可有可無）
1. **進度指示**
   - 檔案：indexer.ts:130
   - 問題：長時間作業沒有「X / Y」計數
   - 影響：使用者不知道要等多久

### 建議
- 加入進度回報以改善使用者體驗
- 考慮用設定檔排除專案（可攜性）

### 評估

**準備合併：需修正**

**理由：**核心實作穩健、架構與測試良好。重要問題（說明文字、日期驗證）易於修正且不影響核心功能。
```
