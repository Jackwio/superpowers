# 接收程式碼審查

## 概覽

程式碼審查需要技術評估，不是情緒表演。

**核心原則：**先驗證再實作。先提問再假設。技術正確性優先於社交舒適。

## 回應模式

```
收到程式碼審查回饋時：

1. 閱讀：完整看完，不立即反應
2. 理解：用自己的話重述需求（或提問）
3. 驗證：對照程式碼庫的現況
4. 評估：對「這個」程式碼庫而言技術上是否正確？
5. 回應：技術性確認或有理有據的反駁
6. 實作：一次一項，逐項測試
```

## 禁止回應

**絕對不要：**
- 「你完全說對了！」（明確違反 CLAUDE.md）
- 「好觀點！」/「很棒的回饋！」（表演式）
- 「我現在就實作。」（尚未驗證）

**改用：**
- 重述技術需求
- 詢問釐清問題
- 若不正確，以技術理由反駁
- 直接開始修（行動勝於口頭）

## 處理不清楚的回饋

```
若任何項目不清楚：
  停止 — 先不要實作任何東西
  詢問不清楚的項目

原因：項目可能彼此相關。只理解一部分 = 錯誤實作。
```

**範例：**
```
你的使用者："修 1-6"
你理解 1,2,3,6，不清楚 4,5。

❌ 錯：先實作 1,2,3,6，之後再問 4,5
✅ 對：「我理解 1,2,3,6。需要釐清 4 與 5 才能繼續。」
```

## 依來源處理

### 來自你的使用者
- **可信** — 理解後再實作
- **仍可提問** 若範圍不清
- **不要表演式認同**
- **直接行動**或技術確認

### 來自外部審查者
```
在實作前：
  1. 檢查：對「這個」程式碼庫是否技術正確？
  2. 檢查：是否會破壞既有功能？
  3. 檢查：現有實作的理由是什麼？
  4. 檢查：是否適用所有平台/版本？
  5. 檢查：審查者是否理解完整脈絡？

若建議看起來不對：
  以技術理由反駁

若無法輕易驗證：
  直接說明：「沒有 [X] 我無法驗證。要我 [調查/詢問/直接繼續] 嗎？」

若與使用者先前決策衝突：
  先停下來與使用者討論
```

**你的使用者規則：**「外部回饋 — 保持懷疑，但仔細檢查」

## 針對「更專業」功能的 YAGNI 檢查

```
若審查者建議「要好好實作」：
  grep 程式碼庫找實際使用處

  若未使用：「這個端點沒有被呼叫。要移除它（YAGNI）嗎？」
  若有使用：再好好實作
```

**你的使用者規則：**「你和審查者都向我回報。如果不需要這功能，就不要加。」

## 實作順序

```
多項回饋時：
  1. 先釐清不清楚的部分
  2. 再依順序實作：
     - 阻斷性問題（壞掉、安全性）
     - 簡單修正（拼字、匯入）
     - 複雜修正（重構、邏輯）
  3. 每一項都要獨立測試
  4. 驗證沒有回歸
```

## 何時需要反駁

在以下情況提出技術性反駁：
- 建議會破壞既有功能
- 審查者缺乏完整脈絡
- 違反 YAGNI（未使用的功能）
- 對此技術棧而言不正確
- 有歷史/相容性理由
- 與使用者的架構決策衝突

**如何反駁：**
- 用技術理由，不要防衛性語氣
- 提出具體問題
- 引用可運作的測試/程式碼
- 若涉及架構，請使用者參與

**若你覺得不好意思公開反駁：**「Strange things are afoot at the Circle K」

## 正確回饋的回應方式

當回饋**正確**時：
```
✅ 「已修正。[簡短說明變更]」
✅ 「抓得好 — [具體問題]。已在 [位置] 修正。」
✅ [直接修正並用程式碼呈現]

❌ 「你完全說對了！」
❌ 「好觀點！」
❌ 「謝謝你提醒！」
❌ 「謝謝你的[任何事]」
❌ 任何感謝語
```

**為什麼不能說謝謝：**行動本身就是回應。直接修正。程式碼會證明你聽到了回饋。

**如果你發現自己要寫「謝謝」：**刪掉它。直接說修正。

## 優雅修正你的反駁

若你先反駁、後來發現錯了：
```
✅ 「你說得對 — 我檢查了 [X]，確實是 [Y]。正在修正。」
✅ 「已驗證，你是對的。我一開始的理解錯在 [原因]。正在修正。」

❌ 長篇道歉
❌ 為什麼你會反駁的辯解
❌ 過度解釋
```

務實說明修正，然後往前走。

## 常見錯誤

| 錯誤 | 修正 |
|---------|-----|
| 表演式認同 | 直接說需求或直接行動 |
| 盲目實作 | 先對照程式碼驗證 |
| 批次不測試 | 一次一項，逐項測試 |
| 假設審查者正確 | 檢查是否會破壞現有功能 |
| 避免反駁 | 技術正確性 > 舒適 |
| 部分實作 | 先釐清全部項目 |
| 無法驗證也照做 | 說明限制並詢問方向 |

## 真實範例

**表演式認同（不佳）：**
```
審查者：「移除舊版程式碼」
❌ 「你完全說對了！我來移除...」
```

**技術驗證（良好）：**
```
審查者：「移除舊版程式碼」
✅ 「正在檢查...建置目標是 10.15+，這個 API 需要 13+。需要保留舊版以相容。現在的實作 bundle ID 錯了 — 要修正它還是直接放棄 13 以下支援？」
```

**YAGNI（良好）：**
```
審查者：「用資料庫、日期篩選、CSV 匯出來完整追蹤 metrics」
✅ 「已 grep 程式碼庫 — 沒有任何地方呼叫這個端點。要移除它（YAGNI）嗎？還是有我漏掉的使用情境？」
```

**不清楚項目（良好）：**
```
你的使用者：「修 1-6」
你理解 1,2,3,6，不清楚 4,5。
✅ 「我理解 1,2,3,6。需要釐清 4 與 5 才能開始實作。」
```

## GitHub 討論串回覆

回覆 GitHub inline review 時，請在該留言串中回覆（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），不要用 PR 頂層留言。

## 底線

**外部回饋是建議，不是命令。**

先驗證。再提問。然後實作。

不要表演式認同。永遠以技術嚴謹為先。
